# Section 09: DOCX Export

## Status: Implemented

## Overview

This section implements the DOCX report generator in `backend/src/features/analysis/export_docx.rs`. It produces a Word document containing the same report structure as the PDF export (section 08): title page, executive summary, coverage charts, per-framework findings tables, priority breakdown, and an appendix with extracted text. Charts are embedded as PNG images generated by the charts module (section 07).

The module exposes a single public function `generate_docx` that accepts analysis data and returns DOCX bytes suitable for HTTP response streaming.

## Deviations from Plan

- **Separate error type:** Instead of sharing `ExportError` from `export_pdf.rs`, a new `DocxExportError` type was defined locally with DOCX-specific variants (no `FontLoading` needed). The PDF module's error type has PDF-specific variants that don't apply.
- **Dynamic chart image sizing:** After code review, `embed_chart_image` reads actual PNG dimensions from the IHDR chunk and computes proportional height, rather than using fixed 400x300 constants. This prevents aspect ratio distortion for variable-height heatmap charts.
- **Table column widths adjusted:** Review identified that the original grid total (10,000 twips) exceeded A4 margins. Reduced to 8,900 twips to fit within standard 1-inch margins.

## Implementation Details

### Function Signature

```rust
use crate::features::analysis::models::{Analysis, AnalysisFindingWithConcept, FindingType};

pub fn generate_docx(
    analysis: &Analysis,
    findings: &[AnalysisFindingWithConcept],
    frameworks: &[(String, String)],  // (framework_id, framework_name)
) -> Result<Vec<u8>, ExportError>
```

The `frameworks` parameter provides the list of matched frameworks with their display names, used for section headings and chart labels.

### ExportError

If not already defined by section 08, define a shared error type:

```rust
#[derive(Debug, thiserror::Error)]
pub enum ExportError {
    #[error("Chart rendering failed: {0}")]
    ChartError(String),
    #[error("Document generation failed: {0}")]
    GenerationError(String),
    #[error("I/O error: {0}")]
    IoError(#[from] std::io::Error),
}
```

### Document Structure

The generated DOCX should mirror the PDF report structure from section 08:

1. **Title page:** Analysis name as a large heading, creation date, status, input type.

2. **Executive summary:** A paragraph with key metrics -- total findings count, gap count, addressed count, number of frameworks matched. Compute these by iterating over the `findings` slice.

3. **Coverage heatmap chart:** Call `charts::render_coverage_heatmap` with framework coverage data (compute coverage as `addressed / total` per framework). Embed the returned PNG bytes into the DOCX as an inline image using `docx_rs::Pic`.

4. **Per-framework sections:** For each framework in `frameworks`:
   - Heading with framework name
   - Radar chart via `charts::render_radar_chart` (if the framework has findings)
   - Findings table with columns: Concept Code, Concept Name, Finding Type, Priority, Confidence, Recommendation

5. **Priority breakdown:** Call `charts::render_priority_chart` with aggregated priority counts. Embed as image.

6. **Appendix:** First 2000 characters of `analysis.extracted_text` (if present), under an "Appendix: Extracted Text" heading.

### Key docx-rs API Usage

The `docx-rs` crate builds documents programmatically:

```rust
use docx_rs::*;

let mut docx = Docx::new();

// Add a heading
docx = docx.add_paragraph(
    Paragraph::new()
        .add_run(Run::new().add_text(&analysis.name).bold())
        .style("Heading1")
);

// Add a regular paragraph
docx = docx.add_paragraph(
    Paragraph::new().add_run(Run::new().add_text("Summary text here"))
);

// Add a table
let rows = vec![
    TableRow::new(vec![
        TableCell::new().add_paragraph(Paragraph::new().add_run(Run::new().add_text("Header"))),
    ]),
];
docx = docx.add_table(Table::new(rows));

// Embed a PNG image
// docx-rs supports adding images via Pic::new with image bytes
```

### Image Embedding

Charts are embedded as inline images. The `docx_rs::Pic` type accepts raw image bytes. Each chart call should be wrapped in a match/if-let to handle `ChartError` gracefully -- if a chart fails to render, log a warning and skip it rather than failing the entire export.

### Rendering to Bytes

The final document is rendered to an in-memory buffer:

```rust
let mut buf = Vec::new();
let cursor = std::io::Cursor::new(&mut buf);
docx.build().pack(cursor)?;
Ok(buf)
```

### Coverage Calculation

To compute framework coverage percentages for the heatmap, group findings by `framework_id` and calculate:

```
coverage = addressed_count / total_findings_for_framework
```

where `addressed_count` includes both `FindingType::Addressed` and `FindingType::PartiallyAddressed` (partial counts as 0.5).

### Priority Aggregation

For the priority bar chart, count findings by priority level across all frameworks:

```rust
let mut priority_counts: BTreeMap<i32, i64> = BTreeMap::new();
for f in findings {
    *priority_counts.entry(f.priority).or_insert(0) += 1;
}
```

Convert to the format expected by `render_priority_chart`: `&[(String, i64)]` where the string is the priority label (e.g., "P1", "P2", "P3", "P4").

## Integration Notes

- This module is called by the export handler (section 10) when the requested format is "docx".
- The export handler passes pre-loaded `Analysis`, `AnalysisFindingWithConcept` vec, and framework name pairs.
- The returned `Vec<u8>` is sent as the HTTP response body with `Content-Type: application/vnd.openxmlformats-officedocument.wordprocessingml.document`.
- Chart rendering calls are synchronous (CPU-bound but fast for small datasets). The export handler wraps the entire `generate_docx` call in `spawn_blocking` since it involves non-trivial computation.
