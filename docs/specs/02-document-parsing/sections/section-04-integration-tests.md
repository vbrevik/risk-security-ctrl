### Building Multipart Request Bodies

The tests need to construct multipart/form-data HTTP request bodies manually since there is no multipart client helper in the test dependencies. Use the following approach:

1. Define a boundary string, e.g., `"----TestBoundary123456"`
2. Construct the body as raw bytes following the multipart format:
   - `--{boundary}\r\n`
   - `Content-Disposition: form-data; name="file"; filename="{filename}"\r\n`
   - `Content-Type: application/octet-stream\r\n`
   - `\r\n`
   - `{file bytes}`
   - `\r\n`
   - `--{boundary}--\r\n`
3. Set the `Content-Type` header to `multipart/form-data; boundary={boundary}`

A helper function `build_multipart_body(filename: &str, data: &[u8]) -> (String, Vec<u8>)` that returns `(content_type_header, body_bytes)` will reduce duplication across tests.

### Database Verification

The `full_pipeline_pdf_upload_to_findings` test verifies data through the API endpoints rather than direct SQL queries, since the test uses `create_test_app()` which returns a `Router` (not the raw database pool). This keeps the test focused on the public contract:

- `GET /api/analyses/{id}` should return the analysis record with `extracted_text` populated
- `GET /api/analyses/{id}/findings` should return a non-empty findings array

If direct DB verification is needed, the test can obtain a pool by modifying `create_test_app()` to also return the `SqlitePool`, or by creating a second helper. This is optional and only recommended if the API endpoints do not yet expose `extracted_text` in their response.

### Test Environment Notes

- The `create_test_app()` function in `backend/tests/common/mod.rs` uses `Config::from_env()` which reads `DATABASE_URL`. For tests, this typically points to a file-based SQLite database. The function runs all migrations and imports ontology data if the `concepts` table is empty.
- Each test gets its own `Router` instance via `oneshot()`, so tests are isolated at the HTTP layer. However, they share the same database file, so test data inserted by one test is visible to others. Use unique analysis names or clean up after tests if ordering matters.
- The `tempfile` crate is already available in `[dev-dependencies]` for generating oversized test data.
- The `tower` crate with the `util` feature (providing `ServiceExt` and `oneshot()`) is already in `[dev-dependencies]`.

### Cleanup Consideration

The upload handler writes files to `uploads/{analysis_id}/`. Integration tests should clean up this directory after running. Either:
- Use `tempfile::TempDir` and override `UPLOAD_DIR` for tests via an environment variable or config field
- Delete the created `uploads/` subdirectory in a cleanup step at the end of each test

The preferred approach is to make the upload directory configurable through `Config` or an environment variable `UPLOAD_DIR`, and set it to a `TempDir` path in tests. This avoids polluting the project directory with test artifacts.
